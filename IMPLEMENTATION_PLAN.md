# План реализации приложения рецептов

## 1) Импорт из Telegram через бота + backend webhook

Обновлённый подход:
- Добавляем бота в приватный канал (лучше с правами администратора).
- Настраиваем webhook **на backend** (`POST /telegram/webhook`), а не на клиентское приложение.
- Backend принимает апдейты Telegram, парсит текст/видео и сохраняет рецепт в БД.
- Веб-приложение при открытии или по кнопке «Обновить» запрашивает backend API (`GET /recipes?since=...`) и подтягивает новые рецепты.

Минимальный контракт для импорта:
- Idempotency по `chat_id + message_id` (чтобы не было дублей).
- Сохраняем `text`, `telegram_file_id`, `imported_at`, `source_chat_id`, `source_message_id`.
- Видео грузим/кешируем через backend, а в UI отдаём стабильный URL.

## 2) Доменная модель рецепта
- `createdAt` (дата добавления)
- `lastCookedAt` (дата последнего приготовления)
- `cookCount` (счётчик приготовлений)
- `isArchived` / `isHidden`
- `isQueued` (флаг «в очередь»)

## 3) Сортировки
Поддержать сортировку по:
- дате создания
- дате последней готовки
- числу приготовлений

Для каждого режима — `asc/desc`.
Правило приоритета: `isQueued=true` всегда выше остальных.

## 4) Архив и временное скрытие
- Быстрое скрытие (`isHidden`) без удаления.
- Отдельный архив (`isArchived`) с возможностью вернуть рецепт обратно.

## 5) PWA для установки на Android
- `manifest.webmanifest`
- Service Worker (offline shell + runtime cache)
- Иконки 192/512 (в т.ч. maskable)

## 6) Готовность к будущему backend-sync
Даже при локальном хранении сразу использовать слой репозитория данных (`RecipeRepository`), чтобы потом заменить local storage на API без переписывания UI-логики.
